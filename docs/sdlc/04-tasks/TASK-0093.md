# TASK-0093: Customer Auth APIs

## Outcome

HTTP APIs for customer OTP request, OTP verification/claim, logout, refresh, and current-profile lookup with tenant/domain enforcement.

## Acceptance criteria

1. Customer auth endpoints resolve tenant context without trusting caller-supplied tenant ids.
2. OTP request sends a short code by email and returns safe challenge metadata.
3. OTP verification issues a customer-scoped session/JWT with role `customer` and tenant/customer claims.
4. Registration or claim binds to an existing SPRINT-019 customer when domain and identity rules allow it.
5. Domain `allow`/`deny` policy is enforced according to tenant auth settings before OTP delivery and session issuance.
6. Cross-tenant customer/session ids return 401/403/404 without revealing foreign data.

## Verification

```bash
go test ./...
go build ./cmd/server
```

## Implementation notes

- Added OTP request/verify, refresh, logout, and `/api/customer/me`.
- OTP request enforces tenant auth settings and SPRINT-019 domain deny/allow rules before mail delivery.
- OTP verify issues role `customer` access tokens plus customer refresh sessions.
