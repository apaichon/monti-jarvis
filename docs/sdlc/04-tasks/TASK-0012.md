# TASK-0012: Auth Middleware + RBAC on Protected Routes

## Outcome

HTTP middleware attaches `AuthContext` to requests; KM write routes require `tenant_admin` or `platform_admin` when auth is enabled.

## Acceptance Criteria

1. Middleware parses `Authorization: Bearer` and populates `user_id`, `tenant_id`, `role` on context.
2. `AUTH_DISABLED=true` skips middleware enforcement (anonymous demo tenant).
3. `AUTH_DISABLED=false`: `POST /api/km/agents/*/documents`, `/reset`, `POST /api/km/seed` return `401` without token, `403` for `customer` role.
4. `platform_admin` may act on any tenant; `tenant_admin` limited to own `tenant_id`.
5. Public routes unchanged: `GET /healthz`, `GET /api/workforce`, `POST /api/chat`, `GET /ws/voice`, `GET /api/km/agents/*` (read).
6. Unit tests for role matrix (admin ok, customer denied, wrong tenant denied).

## Verification

```bash
go test ./internal/auth/... ./cmd/server/...
AUTH_DISABLED=false make restart
# login → upload OK; no token → 401
```
