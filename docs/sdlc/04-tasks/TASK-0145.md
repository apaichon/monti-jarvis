# TASK-0145: Add controlled billing usage reconciliation fixtures

## Outcome

Provide repeatable Postgres, Redis, and ClickHouse fixtures for paid-order/entitlement mismatch, quota divergence, date boundaries, duplicate AI events, and observed/estimated/unavailable usage.

## Acceptance criteria

1. Fixtures can be loaded and reset without mutating shared production-like authorities.
2. Aggregate totals and tenant rows reconcile deterministically across the billing usage API and dashboard.
3. Duplicate projection delivery, missing rates, stale data, and source outage behavior are covered by automated or scripted checks.

## Implementation

[`scripts/sprint32-usage-fixtures.sh`](../../../scripts/sprint32-usage-fixtures.sh) is a scope-guarded `load|reset` harness. It uses an isolated `uat-s31-<suffix>` tenant namespace, transactional Postgres writes in the configured schema, Redis DB 4 with the `monti_jarvis:` prefix, and ClickHouse JSONEachRow inserts using the production fact columns. The primary tenant covers paid/unpaid orders, quota divergence, inclusive dates, duplicate AI delivery, and all measurement states; the secondary tenant is paid but intentionally has no entitlement. Reset deletes only the generated tenant IDs, keys, and ClickHouse facts.

## Review — PASS

Reviewed against DES-0034. The harness is scope-guarded, transactional where applicable, uses the configured Postgres schema/Redis DB 4/ClickHouse database, and resets only generated data. Live fixture-backed UAT passed against the local stack; Postgres, Redis, and ClickHouse cleanup was verified. Duplicate, unavailable/stale, and source-error paths are covered by the runner and focused ClickHouse test. No gRPC/cache production implementation was introduced from design-only DES-0035.
