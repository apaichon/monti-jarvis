# TASK-0036: Payment Provider + Platform Config APIs

## Outcome

`internal/payment/chillpay` exposes `ChillPayClient` (InitPayment, InquiryPaymentStatus, VerifyCallback); platform admins manage gateway config via REST.

## Acceptance Criteria

1. `ChillPayClient` ported from harvest-core pattern: MD5 checksums for init (20 fields + `md5_key`), status inquiry, callback verify.
2. `Provider` interface: `Ping(ctx) error` — ChillPay validates config + optional `InquiryPaymentStatus`; mock always succeeds.
3. `GET /api/platform/payment-gateway` — masked config + `connection_status`; `platform_admin` only.
4. `PUT /api/platform/payment-gateway` — body `{provider, mode, merchant_code, api_key?, md5_key?, base_url, route_no, currency, return_url}`; `callback_url` derived from server base; env `CHILLPAY_API_KEY` / `CHILLPAY_MD5_KEY` override stored secrets when set.
5. `POST /api/platform/payment-gateway/test` — runs `Ping`; `200 {ok:true}` or `502` with error message.
6. Store layer: `GetPaymentGatewayConfig`, `UpsertPaymentGatewayConfig` in `internal/store`.
7. Unit tests: mock provider, config masking, `VerifyCallback` checksum fixture.

## Verification
