# TASK-0036: Payment Provider + Platform Config APIs

## Outcome

`internal/payment` exposes a provider interface with Stripe and mock implementations; platform admins manage gateway config via REST.

## Acceptance Criteria

1. `Provider` interface: `Ping(ctx) error` (credential check); Stripe uses test API; mock always succeeds.
2. `GET /api/platform/payment-gateway` — masked config + `connection_status`; `platform_admin` only.
3. `PUT /api/platform/payment-gateway` — body `{provider, mode, publishable_key, secret_key?, webhook_secret?}`; env `STRIPE_SECRET_KEY` overrides stored secret when set.
4. `POST /api/platform/payment-gateway/test` — runs `Ping`; `200 {ok:true}` or `502` with error message.
5. Store layer: `GetPaymentGatewayConfig`, `UpsertPaymentGatewayConfig` in `internal/store`.
6. Unit tests for mock provider and config masking.

## Verification

```bash
go test ./internal/payment/... ./cmd/server/... -count=1 -run Payment
curl -H "Authorization: Bearer $PLATFORM_TOKEN" http://localhost:8091/api/platform/payment-gateway
curl -X PUT -H "Authorization: Bearer $PLATFORM_TOKEN" -H "Content-Type: application/json" \
  -d '{"provider":"mock","mode":"test"}' http://localhost:8091/api/platform/payment-gateway
curl -X POST -H "Authorization: Bearer $PLATFORM_TOKEN" http://localhost:8091/api/platform/payment-gateway/test
```
