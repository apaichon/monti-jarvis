# TASK-0032: KYC Approve/Reject Lifecycle

## Outcome

Approve/reject runs in a single transaction; approved tenants pass `RequireKMWrite` and related active-tenant checks.

## Acceptance Criteria

1. Approve transaction updates atomically: `tenants.status=active`, `tenant_registrations.status=approved`, `tenant_kyc_profiles.status=approved`, reviewer fields set.
2. Reject transaction updates: `tenant_registrations.status=rejected`, `rejection_reason`, reviewer fields; `tenant_kyc_profiles.status=rejected`; tenant stays `pending_kyc`.
3. Double-approve or approve-after-reject returns `409`.
4. `internal/auth` middleware: `tenant_admin` on `active` tenant can `POST /api/km/*`; `pending_kyc` still `403`.
5. Unit test covers approve path and KM write gate regression (`middleware_test.go` or store test).

## Verification

```bash
go test ./internal/auth/... ./cmd/server/... -count=1
# After approve: tenant_admin POST /api/km/agents/{slug}/documents → not 403
```
