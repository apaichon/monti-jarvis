# TASK-0068: Tenant KM API

## Outcome

First-class **tenant_admin** APIs for KM lifecycle, always scoped to JWT tenant.

## Acceptance Criteria

1. Endpoints (all `RequireTenantAdminActive`):

   | Method | Path | Purpose |
   | --- | --- | --- |
   | `GET` | `/api/tenant/km/scopes` | Allowed scopes: general, billing, technical |
   | `GET` | `/api/tenant/km/agents` | Agents available to tenant + doc counts |
   | `GET` | `/api/tenant/km/agents/{agent_id}/documents` | List docs for agent |
   | `POST` | `/api/tenant/km/agents/{agent_id}/documents` | Multipart upload + `scope` form field |
   | `PATCH` | `/api/tenant/km/documents/{id}` | Update `km_scope` (and re-index if required) |
   | `DELETE` | `/api/tenant/km/documents/{id}` | Delete document (0067 cascade) |
   | `POST` | `/api/tenant/km/agents/{agent_id}/reset` | Clear agent KM for tenant |

2. Tenant isolation: cannot read/write other tenants' documents.
3. Invalid agent_id → 400; missing file → 400; unknown doc → 404.
4. Wire **S13** KM rate limit + max documents on upload.
5. Prefer reusing `internal/km` ingest; do not bypass embeddings.
6. Handler/store tests for happy path + isolation.
7. Keep legacy `/api/km/*` for platform/ops (document dual surface in API spec).

## Verification
