# TASK-0095: Customer Auth UX

## Outcome

Customer portal email OTP login/account state that preserves public no-auth conversation access and passes authenticated customer context into chat/voice flows.

## Acceptance criteria

1. Customer portal can request OTP by email, verify OTP, show authenticated account state, and logout.
2. Existing public no-auth conversation remains available when customer auth is disabled or not required.
3. Authenticated chat/voice calls include tenant/customer/tier/group context for quota and scope decisions.
4. Customer cannot inspect or switch to another tenant's customer identity.
5. UX handles disabled auth, denied domain, inactive customer, OTP delivery failure, expired OTP, and expired session states.

## Verification

```bash
npm --prefix apps/customer-web run check
npm --prefix apps/customer-web run build
```

## Implementation notes

- Added optional customer OTP sign-in/account panel to the public caller desk.
- Customer access tokens are attached to chat and call API requests.
- No-auth conversation access remains available when the customer chooses not to sign in.
