# TASK-0090: Customer Assignment and Integration Contract

## Outcome

Customer records bind safely to SPRINT-018 tiers/groups and expose deterministic source/external-id upsert behavior for future connectors.

## Acceptance criteria

1. Validate tier/group ownership against the JWT tenant on create, update, and import.
2. `(tenant_id, source, external_id)` is the primary integration idempotency key; normalized email is the fallback for `source=manual|csv`.
3. Reimport updates mutable profile/assignment fields while preserving customer id and audit history.
4. Domain defaults apply only when an explicit CSV/API tier/group is absent.
5. Emit or log a stable `customer.upserted` integration event contract without delivering external webhooks in SPRINT-019.
6. Tests cover explicit assignment precedence, defaults, and cross-tenant rejection.

## Verification

```bash
make test
# import same source/external_id twice; assert one customer id
```

## Implementation evidence

- Source/external-id matching precedes normalized-email fallback and returns a conflict when identities disagree.
- Explicit assignments take precedence; matching domain defaults fill only an unassigned customer.
- Same-tenant tier/group validation runs for manual, API, dry-run, and commit paths.
- Successful upserts publish the non-fatal `monti.customer.upserted` NATS contract when available.

## Review — PASS

Idempotency, assignment precedence, atomic batch behavior, tenant isolation, and non-fatal integration events match workflow §57–58 and DES-0022.
